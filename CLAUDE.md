# CLAUDE.md

Architectural invariants and ways of working for this repo. Lean by design. Read in full at session start. For match-engine internals see **`docs/match-engine.md`**; for the season/career engine see **`docs/game-engine.md`**; for the transfer system roadmap see **`docs/transfer-system.md`**; Visual & interaction conventions are in docs/DESIGN.md. Read it before touching CSS or UI components.
---

## 1. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them — don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

## 2. Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.
- Do not deep-clone data structures just to pass them to `JSON.stringify()` (e.g., save payloads). `JSON.stringify` naturally traverses objects without mutation; explicitly allocating deep clones beforehand causes severe GC churn.
- Avoid heavy functional object pipelines (e.g., `Object.fromEntries(Object.entries(x).map(...))`) inside high-frequency engine loops. A simple `for` loop populating a single object is faster and avoids severe GC churn.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

**Resolve constant arithmetic at write-time, not runtime.** If a formula applies a fixed multiplier or offset to a literal or random range, collapse it. `rng(0.5, 1.5) * 8` → `rng(4, 12)`. The result should express the actual value used, not the derivation of it.

## 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it — don't delete it.
- Remove imports/variables/functions that YOUR changes made unused.

The test: every changed line traces directly to the user's request.

**Restructuring a live type doesn't restructure its snapshots.** A frozen log row, an event-bus payload, or a replay event has schema lifetime independent of the live state it was copied from. `GameEvent.ballX/ballY`, `PenaltyContext.*`, and `MatchEvent` payload fields stay scalar even when their source moved into `state.ball`/`state.clock`. The test: would renaming break replay, an existing log entry, or a downstream consumer that already serialised the old shape? If yes, leave it alone.

**Namespacing is not decoupling.** `state.phase.breakdownMod` has identical coupling properties to `state.breakdownMod`. Before drafting a "decouple" refactor, name the specific coupling smell and verify the proposed shape actually removes it.

## 4. Module Boundaries

**Split before god objects form. Don't wrap clean primitives.**

- Split a file when it accumulates multiple unrelated responsibilities. Name each piece by what it does (Coordinator, Controller, Router, Handler). Don't wait for thousands of lines.
- Push back on abstractions that wrap already-clean primitives. Don't wrap a typed state object in a "store"; don't wrap a typed pub/sub singleton in a "dispatcher".
- Prefer pure functions over methods when state can be passed directly. `FieldPosition` helpers take `state: MatchState` as an argument; they are not closures threaded through a deps interface.
- Use constructor DI for classes whose methods share the same deps (`PenaltyHandler`, `CardHandler`, `ClockController`). Use module-level functions for pure helpers (`FieldPosition`, `PhaseRouter`).
- Extract a shared utility the moment a second module needs it, not before.
- Refactor incrementally. One cohesive split per commit; each commit must build clean and preserve behaviour.
- A module-boundary change is an engine change — update the matching engine doc in the same commit (`docs/match-engine.md` for `src/engine/` work, `docs/game-engine.md` for `src/game/` work).
- **Navigation goes through `screenRouter.show(id)`** (`src/ui/ScreenRouter.ts`). Screen modules never poke `document.getElementById('…').style.display` directly; they accept `onForward`/`onBack` callbacks from `main.ts`. Adding a screen: (1) add the id to the `SCREENS` map in `ScreenRouter.ts`, (2) add a `<div id="…">` to `index.html`, (3) add a flat navigation handler in `main.ts`.
- **In-season screens (`HubScreen`, `FixtureListScreen`, `LeagueMenuScreen`, `LeagueTableScreen`, `TeamStatsScreen`, `PlayerStatsScreen`, `RoundResultsScreen`, `EndOfSeasonScreen`, `RenewalsScreen`, `TransferMarketScreen`, `RolloverScreen`, `ContractsScreen`, `SquadManagementScreen`, `TrainingScreen`) are initialised once per page lifetime, not per game and not per navigation.** `initInSeasonScreens()` in `main.ts` is gated by an `inSeasonInited` closure flag, so the second call (e.g. New Game → Home → Continue) is a no-op. This is load-bearing: each screen registers `eventBus.on('game:*')` subscriptions at init time without an unsub, so without the gate every back/forward (or game switch) would duplicate handlers and leak. Subsequent visits use bare `screenRouter.show(id)`. **Each screen takes a `getGameEngine: () => GameCoordinator` getter** (not the engine reference directly) and reads `getGameEngine().getState()` on every render, so when `main.ts` reassigns `gameEngine` for a new game the live engine reaches every screen without re-init. Capturing the reference at init time would freeze the screens to the first game — the getter is the seam that makes the once-per-lifetime init contract safe across engine swaps. **Hub is the top of the in-season stack** — no back arrow; the Settings cog is the exit route to Home. Fixture-list, league-table, contracts, and squad-management back-buttons all return to Hub.
- **Dual-mode in-season screens (post-match Continue flow).** `RoundResultsScreen` and `LeagueTableScreen` each expose a module-level setter — `showRoundResults(round, onContinue)`, `showLeagueTablePostMatch(onContinue)` — that the navigation handler calls *before* `screenRouter.show(id)`. The setter updates closure state and invokes a stored `renderImpl` so the forward "Continue → next" CTA is visible immediately, replacing the back arrow. The hub-entry path doesn't call the setter and gets the back-arrow render. Mode is cleared on the forward click. New dual-mode screens should follow the same pattern; don't reach for a global store and don't re-init the screen. `EndOfSeasonScreen` (one-shot, `showEndOfSeason(onContinue)`), `RenewalsScreen` (one-shot, `showRenewals(onContinue)` — onContinue receives the user's `Record<offerId, 'renew'|'release'>` decisions), `TransferMarketScreen` (one-shot, `showTransferMarket(onContinue)`), and `RolloverScreen` (one-shot, `showRollover(events, onContinue)`) use the same setter pattern; all four are only reachable from the post-match chain when `game:seasonComplete` has latched. **`TrainingScreen` is dual-mode**: `showTrainingPostMatch(onContinue)` (post-match flow — Continue applies one week of training league-wide via `applyTrainingWeek`) and `showTrainingMidweek(onBack)` (Hub-tile entry — Back persists the plan via `setPlayerTrainingPlan` without executing).
- **`MatchCoordinator` owns its event-bus subscriptions and must be destroyed.** Constructor captures unsubs in `busUnsubs[]`; `destroy()` runs them, cancels the tick timer, and clears the run flag. `main.ts` calls `engine.destroy()` after the match-result overlay is dismissed. The **`silent: true`** constructor option (used by `src/game/simulateFixture.ts` for headless AI fixtures inside `recordPlayerMatchResult`) suppresses every `engine:*` emit except `engine:finished`, skips the constructor's UI-event subscriptions, short-circuits `PenaltyHandler` modal prompts to `high_ball` / `kick_for_goal` defaults, collapses `CardHandler`'s 3-tick TMO narrative to a single inline application (RNG calls happen in identical order so silent and live agree; the `tmo_intervenes` + `tmo_decision_*` announcements still fire in the collapsed tick so the events log keeps a faithful audit trail for telemetry), and auto-picks red_20 replacements by position group. `ClockController` and `FatigueAccumulator` take the same flag — every engine orchestrator that emits to the bus must gate on it.
- **`AITacticalDirector` is a pure, RNG-free module owned by `MatchCoordinator`** (`src/engine/AITacticalDirector.ts`). Called once per tick from `tick()` *before* `resolvePhase()` so the new tactics affect the same tick that triggered them. Constructor takes `humanSide: TeamSide | undefined`; in a live match `MatchCoordinator` passes the player's side (the director never touches it), in silent fixtures it passes `undefined` (both teams adapt). All tactic changes flow through `TACTICS_UPDATED` via `applyMatchEvent` — same mutation boundary as the in-match tactics modal. Tuning constants live in `src/engine/balance/aiDirector.ts`.
- **`AISubstitutionDirector` is the sibling sub-making orchestrator** (`src/engine/AISubstitutionDirector.ts`) — pure, RNG-free, owned by `MatchCoordinator`, called once per tick immediately after `AITacticalDirector.evaluate()` and before `resolvePhase()`. Same `humanSide` gate (live match → AI side only; silent fixtures → both teams). From `AI_SUBS_VALUES.earliestSubMinute` (50') onwards it loops while a starter sits at or below `AI_SUBS_VALUES.fatigueThreshold` (60%) and the bench has an exact-position or position-group replacement; each swap is routed through `MatchCoordinator.substitute()` so the `SUBSTITUTION_APPLIED` mutation seam + commentary + silent-mode gate are identical to a manager sub. Bulk subs are allowed — the loop is bounded only by bench depletion / no eligible candidate, matching real-world coaches clearing the bench in a single window around 50–55'. Tuning constants live in `src/engine/balance/aiSubs.ts`.
- **`CardHandler` runs before `PenaltyHandler` on every penalty.** Owned by `MatchCoordinator` like the other orchestrators. `evaluateNewPenalty()` rolls TMO (per-offence trigger rate from `OFFENCE_SPEC` in `balance/discipline.ts` — `high_tackle` and `dangerous_cleanout` sit at 90%, the others at 0%) or team-22-rule (4th defensive penalty in own 22 → forced yellow); on a TMO trigger it transitions phase to `TmoReview` and the next 3 ticks drive narrative (`advanceTmoReview()`) with the clock frozen via `ClockController.advanceMinute` short-circuit. `scanSinBinReturns()` runs every non-TMO tick and fires `SIN_BIN_RETURNED` / `RED_20_EXPIRED`; red_20 expiry triggers `MatchCoordinator.runForcedSubstitution(player, side, 'red_20')` (forced sub modal for human side, position-group auto-pick for AI / silent). All mutations flow through `CARD_ISSUED` / `SIN_BIN_RETURNED` / `RED_20_EXPIRED` / `TEAM_PENALTY_22_RECORDED` / `TEAM_22_WARNING_ISSUED` / `TMO_REVIEW_*` `MatchEvent` variants. Resolvers and selectors filter team rosters through `onFieldPlayers(team, state, side)` (`src/engine/FieldPosition.ts`) so a sin-binned forward genuinely weakens the scrum / lineout / breakdown (no separate flag on `Player`). Full breakdown in `docs/match-engine.md` § "Cards (Yellow / Red 20 / Red full)".
- **`KickDecisionDirector` is the unified kick-or-carry decision module.** (`src/engine/KickDecisionDirector.ts`) Replaces the three independent kick gates that lived at the top of `PhasePlay` / `FirstPhase` / `KickReturn` AND the dedicated Breakdown slow_ball → BoxKick gate that lived in `BreakdownEvent`. Single decision tree reads `attackingGamePlan` + field zone + `state.lastBallQuality` (set by Breakdown clean/slow outcomes); rolls kick/carry; picks one of four families (`clearance` / `territory` / `fifty_22` / `attacking`); picks kicker (#9 → BoxKick phase / #10 → TacticalKick phase); for clearance and attacking, pre-rolls the sub-choice (`long_and_on` vs `long_and_off`, `cross_field` vs `grubber`). The family + sub-choice flow through `state.pendingKick` to the kick handler via `KICK_INTENT_SET`. Resolvers branch their math on `pendingKick`: `BoxKickResolver` adds a `goes_to_touch` outcome for `long_and_off` clearance; `TacticalKickResolver` adds `resolveFiftyTwentyTwo` (defender-backfield-gated) and `resolveAttackingKick(cross_field | grubber)` branches. Tuning lives in `src/engine/balance/kickDecision.ts` + `src/engine/balance/kicking.ts`. Full breakdown in `docs/match-engine.md` § "Kick Decision Director".
- **`runForcedSubstitution(player, side, reason)` is the shared red_20 / injury sub flow.** Owned by `MatchCoordinator`. `reason: 'red_20' | 'injury'` picks the announcement keys; the modal payload + auto-pick fallback are identical for both paths. Called once per expired red_20 (from `scanSinBinReturns()`) and once per pending injury per tick (from `collectPendingInjurySubs()` — walks `state.cards.injured` for players still occupying a field slot). `SUBSTITUTION_APPLIED` strips both `cards.sentOff` and `cards.injured` so the replacement player isn't filtered out by `onFieldPlayers`. In-match injuries are pushed onto `cards.injured` by `PLAYER_INJURED_IN_MATCH` from `OpenPlayEvent.rollMatchInjury` at the end of `handlePhasePlay`; severity + duration are NOT rolled in-match — `Player.pendingInjuryKind` is read at teardown and the `rngTransfer`-driven `PLAYER_INJURED` season event fires inside `GameCoordinator.recordPlayerMatchResult`. Full breakdown in `docs/match-engine.md` § "Injuries" and `docs/game-engine.md` § "Injuries".

## 5. Mutation Boundaries

**State mutation flows through one function. Don't sneak in a direct write.**

- All writes to `MatchState`, `player.matchStats`, `player.fatiguePct`, `player.currentStats`, and `player.rating` go through `applyMatchEvent(state, event)` in `src/engine/applyMatchEvent.ts`. No exceptions, including `state.events.push(...)`.
- **`applyMatchEvent` runs `assertInvariants(state)` after every event** (`src/engine/invariants.ts`). It throws if score/possession/phase/ball/clock or any player's `fatiguePct`/`rating`/`currentStats` strays outside its legal range. Always-on, not env-gated — cost is O(matchday squad) per mutation, negligible against the per-tick work already done. Adding a new mutation kind that touches one of these fields: extend the invariant check too if it could push a value off the asserted range.
- Phase handlers in `src/engine/events/` are read-only over state: they read, compute, build a `MatchEvent[]`, and return it on `PhaseResult.events`. `PhaseRouter.resolvePhase()` applies the queue then composes the outgoing `GameEvent`.
- Orchestrators (`MatchCoordinator`, `ClockController`, `PenaltyHandler`, `CardHandler`) call `applyMatchEvent` directly for non-phase mutations (clock, half-time, penalty choices, cards, sin-bin lifecycle, TMO review steps, substitutions, tactics, fatigue, commentary log).
- Use **domain-meaningful** event names (`TRY_SCORED`, `KNOCK_ON`, `CARRY_RESOLVED`, `LINEOUT_RESOLVED`) — not primitive setters. Narrow exception: structural setters (`BALL_REPOSITIONED`, `PHASE_CHANGED`, `BREAKDOWN_MOD_SET`, `POSSESSION_SWAPPED`, `POSSESSION_SET`) where the domain has no single name.
- **`state.phase` is the sole source of truth for the current phase.** All transitions go through `PHASE_CHANGED` via `applyMatchEvent`; there is no separate state-machine class. The valid-transition table in `docs/match-engine.md` is documentary, not enforced at runtime.
- **`PhaseContext` is the minimal closure passed to handlers** — `{ state, attackTeam, defendTeam, randomPlayer, pickPlayer, draftEvent, kickOffStrategy }`. Field-position helpers (`attackDir`, `inOwn22`, `isTryScoredAt`, …) are pure functions in `src/engine/FieldPosition.ts` and take `state` as an argument — handlers import them directly rather than threading them through the context.
- Adding a new mutation kind: one variant in the `MatchEvent` union (`src/types/matchEvent.ts`), one branch in `applyMatchEvent`. The exhaustive `default: const _: never = event;` catches missing branches at compile time.
- Adding a new player stat: extend `PlayerMatchStats` + `zeroMatchStats()` (both in `src/types/player.ts`, co-located) + the relevant domain event's apply branch — never push a raw `player.matchStats.X++` into a handler.
- `eventBus.emit` calls are **pure UI side effects** and are NOT part of the mutation boundary. They live in orchestrators alongside `applyMatchEvent` calls, not inside `applyMatchEvent` itself.
- Derived state (`computeRating`, `computeFatigue`) lives in pure helpers; writes still flow through dedicated `MatchEvent` variants (`RATINGS_RECALCULATED`, `FATIGUE_APPLIED`).
- **Do not redundantly clone payloads inside `applyMatchEvent`.** If an orchestrator has already allocated a fresh object for the event payload, assign it by reference (`event.player.currentStats = event.newCurrentStats`) instead of spreading it again (`{ ...event.newCurrentStats }`). Redundant clones multiply GC pressure.
- **Pre-match jersey assignment vs in-game substitution are different operations.** Pre-match (`PreMatchScreen.assignStartingJersey`) reassigns `squadNumber` AND `id` by slot — starting XV wears 1–15, bench wears 16–23. In-game substitution (`SUBSTITUTION_APPLIED`) reassigns ONLY `id`/`position`/`x`/`y`; the substituting player keeps their bench `squadNumber`. Both flows assume `squadNumber` is unique across `team.players ∪ team.bench`.
- **Season-scope mutations have their own boundary: `applySeasonEvent(state, event)`** in `src/game/applySeasonEvent.ts`, operating on `GameState` (`src/types/gameState.ts`). `GameCoordinator` is the only caller. `SeasonEvent` variants are domain-meaningful and the same `default: const _: never = event;` exhaustiveness contract applies. **`applySeasonEvent` runs `assertSeasonInvariants(state)` after every event** (`src/game/seasonInvariants.ts`) — sibling of the match-side tripwire. It throws if any rosterId in `clubs.squad` / `freeAgents` / `pendingMoves` / `market.offers` is orphaned (not in `state.career.roster`), if a rosterId appears in two clubs' squads simultaneously or in both a squad and `freeAgents`, if any club has more than one marquee, if `nextRosterId` is not strictly greater than every used rosterId, if a wage / season-stat counter goes negative, if `standings.played !== W + D + L`, or if a set-piece won total exceeds attempts. Adding a new mutation kind that touches one of these fields: extend `assertSeasonInvariants` too if it could push a value off the asserted range. Always-on, not env-gated — cost is O(roster + clubs) per call, negligible against the per-event mutation work already done.

## 6. Randomness Boundary

**All randomness flows through `src/utils/rng.ts`. Never call `Math.random()` directly in engine code.**

Four isolated mulberry32 streams:
- `rng(min, max)` — outcome stream; every in-play roll. Reset by `setMatchSeed(seed)` (called from the `MatchCoordinator` constructor).
- `rngForm()` — form stream; player form modifier at `initPlayer()`. Reset by `setMatchSeed`.
- `pickRandom(arr)` / `commentaryChance(pct)` — commentary stream; flavour-text sampling. Reset by `setMatchSeed`.
- `rngTransfer(min, max)` / `rngTransferRaw()` — career stream; contract seeding, aging-curve noise, retirement rolls, future transfer / persona generation. Reset by `setCareerSeed(seed)` (called from `GameCoordinator.newSeason` / `fromSave`) — independent of the match seed so a per-fixture derivation cannot perturb season-scope outcomes.

Streams are independent — adding a commentary line cannot shift outcome rolls; adding a transfer event cannot shift a match. Pick the matching stream when adding a randomness consumer. A match with a given seed is fully reproducible; a career with `(playerTeamId, rootSeed)` is fully reproducible across multiple seasons. Full breakdown in `docs/match-engine.md` "Determinism (Seeded RNG)" and `docs/game-engine.md` "Determinism".

## Balance constants

## Architecture

**Engine ↔ UI contract.** The engine never imports from UI; UI never calls engine methods directly **except** `SimController` (Play/Pause/Speed). All communication goes through the typed pub/sub singleton at `src/utils/eventBus.ts`. Within a single tick, `engine:event` is emitted **before** `engine:stateChange` — UI state caches from the prior tick are always valid by the time an event arrives. Engine event table and subscribers: `docs/match-engine.md` § "UI Event Bus Contract". UI subscriptions registered at startup are permanent; one-shots are explicitly unsub'd (e.g. `CommentaryFeed`'s team-colour cache).

**Game engine ↔ UI contract.** Analogous, season-scope. `GameCoordinator` (`src/game/`) emits five `game:*` events on the same bus: `game:initialized` (after `newSeason` / `fromSave`), `game:fixtureRecorded` (once per result — the player's match plus every headless AI fixture of that round), `game:weekAdvanced` (after the round completes and the calendar steps forward), `game:seasonComplete` (latched in `main.ts` so the post-match Continue chain reroutes through EndOfSeason → Renewals → TransferMarket → Rollover), and `game:seasonRolledOver` (after `rollSeason()` applies aging + retirements + transfer activations + academy intake + new fixtures). Hub, FixtureList, LeagueTable, RoundResults, EndOfSeason, Rollover, and Contracts all subscribe / render reactively. Headless AI fixtures inside `recordPlayerMatchResult` run silent `MatchCoordinator` instances, so they emit no `engine:*` UI noise — but they still emit `game:fixtureRecorded` so the league table updates live.

**`state.engine.humanSide` is the canonical source for which side the manager picked.** Set once in `initMatchState` from `MatchCoordinator`'s `humanSide` opt; never mutated. UI surfaces that order team-specific data "player first" — `StatsPanel`'s live player rows and detailed table, `MatchResultScreen`'s ratings cards, `PreMatchScreen`'s versus header and tabs — read it from state rather than threading the side through closures or props. Silent fixtures default it to `'home'`; the UI never sees those.

**PreMatchScreen leads with the player's team visually** — left versus slot + first/active tab — regardless of home/away. Engine identity (home team is still home) is preserved; only the on-screen order changes. The screen takes `gameEngine` and reads `league.results` + `league.standings` at render time for its form pins, H2H, league-position and odds tiles. No `game:*` subscriptions — it's a one-shot per match. Derivation helpers live in `src/game/teamStats.ts` (`recentForm`, `headToHead`, `matchSpread`); the screen consumes them as pure functions.

**Home advantage is engine-level, not display-only.** `HOME_ADVANTAGE` (`src/engine/balance/homeAdvantage.ts`) is consumed in two places: the carry resolvers (`OpenPlay`/`FirstPhase`/`KickReturn`) and `BreakdownEvent`, via the `homeEdge(state, mod)` helper that splits a flat `mod` into `{ attack, defend }` based on `state.possession`. The pre-match SPREAD tile re-exports the same `HOME_ADVANTAGE.spreadPts` through `src/game/teamStats.ts` so the prediction and the simulation share one constant — moving the dial in balance shifts both. Calibration target is ~57% home win-rate, validated via `npm run telemetry` (which now reports a Home Advantage section).

**Commentary is data, not text.** The engine populates `NarrationDescriptor` on every `GameEvent`; `CommentaryFeed.ts` calls `renderNarration(event)` to produce strings. CommentaryFeed is initialised once at startup and listens permanently; **silencing during a headless AI fixture is the engine's job, not the UI's** — set `silent: true` on the `MatchCoordinator` (see Section 4).

**Attack direction.** Home attacks toward x=100 in the first half, toward x=0 in the second. Teams swap ends only at half-time, never on turnovers. All `ball.x` reasoning must go through the pure helpers in `src/engine/FieldPosition.ts` — they factor in `state.clock.halfTimeDone`. Snapshot DTOs (`GameEvent`, `PenaltyContext`, `MatchEvent` payloads) keep flat `ballX`/`ballY` scalars; see Section 3.

**Tactics.** Seven dimensions in `TeamTactics` (`src/types/team.ts`) — `attackingGamePlan`, `attackingStyle`, `attackingBreakdown`, `defendingBreakdown`, `backfieldDefence`, `defensiveLine`, `offloadStrategy`. The managed team can change all seven mid-match via the tactics modal. The AI side opens each match on its team's authored `suggestedTactics` (from `src/data/team-*.json`, surfaced on `RawTeamInput.suggestedTactics`) — *not* `DEFAULT_TACTICS`, which is now only the fallback when no authored value exists. Mid-match the AI is adapted by `AITacticalDirector` (see Section 4): inside the final 15 minutes a score gap of ≥ 8 flips it into `AI_INTENT_CHASING` (trailing) or `AI_INTENT_PROTECTING` (leading); otherwise it sits on the baseline. The director never proposes tactics for the human side. Kick-off strategy is per-kick (not a standing tactic) — managed team picks via modal, AI defaults to `high_ball`. Effects, probability tables, and tactic-note triggers: `docs/match-engine.md` "Tactics: who picks what" + "Carry Phases" + "Tactical Commentary".

**Cards.** Three kinds — `yellow` (10-min sin-bin, returns), `red_20` (20-min off, no return but team may sub from bench), `red_full` (permanent sent off; no trigger today). Two trigger paths live: TMO (per-offence trigger rate from `OFFENCE_SPEC` in `balance/discipline.ts` — `high_tackle` and `dangerous_cleanout` both at 90%; on hit → 3-tick narrative with frozen clock → 25/65/10 no-card/yellow/red_20 outcome from the global `TMO` weights), and the team-22 rule (4th defensive penalty in own 22 → forced yellow on the offender, captain warning at 3). Adding a new TMO-eligible offence is a one-line edit to the `OFFENCE_SPEC` registry — no `CardHandler` change. Owned by `CardHandler` (Section 4); state lives at `state.cards` (sinBin + sentOff per side, plus team-22 counters and the one-shot warning flag) and the optional `state.tmoReview` during a review. All mutations through `applyMatchEvent` via dedicated `CARD_ISSUED` / `SIN_BIN_RETURNED` / `RED_20_EXPIRED` / `TEAM_PENALTY_22_RECORDED` / `TEAM_22_WARNING_ISSUED` / `TMO_REVIEW_*` events. Sin-binned players stay in `team.players`; resolvers exclude them via `onFieldPlayers(team, state, side)` (`src/engine/FieldPosition.ts`), so the scrum / lineout / breakdown weakens naturally when a forward is off (`ScrumResolver.packScore` is a **sum**, not an average, to make this load-bearing). For a missing back, `SHORT_HANDED.missingBackDefendPenalty` (in `balance/discipline.ts`) feeds the same `defendMod` term as `backfieldLineBreakPenalty`. Tuning constants: `balance/discipline.ts` (TMO trigger + outcome weights, sin-bin durations, team-22 thresholds, missing-back penalty). Full breakdown in `docs/match-engine.md` § "Cards (Yellow / Red 20 / Red full)".

**Mauls.** `MatchPhase.Maul` is a single-resolve phase reachable only from a clean lineout catch, gated by `MAUL_GATE` in `balance/maul.ts` — zone-driven (own half 0%, opp half 5%, opp 22 35%, inside opp 10m 80%) and modified by the attacking team's `attackingStyle` (`keep_it_tight` +20pp, `wide_wide` -20pp). The handler (`src/engine/events/MaulEvent.ts`) uses `availableForwards` for both packs and resolves through `MaulResolver` — a SUM-based packScore (so a sin-binned / sent-off / injured forward genuinely weakens the maul, mirroring scrum) plus a two-stage outcome: stage 1 (strength margin) decides `maul_won` vs `maul_held`; stage 2 (on a maul_won, the defender may collapse cynically — probability scales with pressure margin + inverse defender discipline) decides `maul_collapse_penalty`. Won mauls gain 5-10m (~10% chance of a 15-25m long drive), reposition the ball forward via `BALL_REPOSITIONED`, and transition to `TryScored` (hooker credited as `primaryPlayer`) if the projected ball crosses the line, otherwise `FirstPhase`. Held mauls flip possession to a turnover scrum. Collapse penalties emit `PENALTY_AWARDED { offence: 'maul_collapse', ... }`; CardHandler then runs a zone-scaled direct yellow check (`MAUL_COLLAPSE_YELLOW`: 5% / 30% / 70% by zone, no TMO narrative — `maul_collapse` keeps `tmoTriggerPct: 0` in OFFENCE_SPEC). Team-level stats live at `state.stats.mauls[side]` (won + collapse_penalty count) and `state.stats.maulMetres[side]` (gained ground). No per-player maul counters today. Full breakdown in `docs/match-engine.md` § "Maul".

**Design system.** `docs/DESIGN.md` is the single source of truth for every colour, font, spacing, and component pattern. CSS custom properties in `style/main.css` `:root` — no hardcoded hex except primary CTA green (`#007a2a` / `#009434` active / `#006622` pressed) and team identity colours injected inline from team JSON.

**Team data.** `src/data/team-*.json` — all 10 Gallagher Premiership clubs (Bath, Bristol, Exeter, Gloucester, Harlequins, Leicester, Newcastle, Northampton, Sale, Saracens). The JSON shape is typed as `RawTeamInput` in `src/types/teamData.ts` (deliberately neutral — both the match engine and the game engine import it from there; nothing engine-internal owns the data contract). Each file has `players` (15 starters, id 1-15), `bench` (8 matchday subs, id 16-23), and `squad` (the rest of the senior roster, id 24+). Each player carries `firstName`, `lastName`, `dob` (nullable ISO), `nationality`, `position` (generic — `Prop`, `Hooker`, `Lock`, `Flanker`, `Number 8`, `Back Row`, `Scrum-Half`, `Fly-Half`, `Centre`, `Wing`, `Fullback`, `Utility Back` — see `Position` in `src/types/player.ts`), and 12 `baseStats` on a 1–100 scale. `MatchCoordinator.initPlayer` copies `baseStats` to `currentStats` at match start; `StaminaSystem.computeFatigue` drives `currentStats` over the match via `FATIGUE_APPLIED`. `baseStats` is never modified **in-match** (but see the spawn pipeline below). Player full names are unique league-wide — `CommentaryFeed`'s colourisation pass relies on this. **Source of truth is `docs/team-data.md`** — to regenerate the JSONs deterministically after editing team-data.md, run `node scripts/generateTeamJsons.mjs`. The doc's `### Squad (2025-26)` block is tiered into five sub-tables — `Starting XV — Forwards/Backs`, `Bench`, `Wider squad — Forwards/Backs` — and each player's tier **is** their authored role. The 12 `baseStats` are manually authored in the right-most columns of these markdown tables, replacing the old deterministic generation formulas. The generator hard-errors if Starting XV ≠ 15, Bench ≠ 8, or a `### Star players` entry isn't in the Starting XV tables.

**Position is generic, not split.** The engine never branches on `player.position` for match logic — scrum/lineout/breakdown/first-phase/fatigue/rating-bonuses all key off `player.id` (jersey 1-15). `position` is read only by `PLAYER_OVERALL_WEIGHTS` / `IRRELEVANT_STATS` lookups, UI display, and the substitution position-copy. Don't reintroduce Loosehead/Tighthead, Left/Right Lock, Blindside/Openside Flanker, Inside/Outside Centre, Left/Right Wing splits — the generic form is the data contract. **Slot constants live in `src/engine/Slot.ts`** — `SLOT.SCRUM_HALF`, `SLOT.FLY_HALF`, `SLOT.HOOKER`, etc. (1-15), plus `isForwardSlot` / `isBackSlot` / `isFrontRowSlot` / `isBackRowSlot` / `isMatchdaySlot` predicates and `STARTING_XV_MAX` / `MATCHDAY_MAX` bounds. Engine code consumes these instead of bare numeric literals; do not reintroduce raw jersey-number `=== 9` / `>= 6 && <= 8` style checks in new code. **Specialist-slot lookups for kickers / receivers go through `pickKicker(team, state, side)` / `pickScrumHalf(...)` / `pickFullback(...)` in `src/engine/FieldPosition.ts`** — each consults `onFieldPlayers` first and degrades through positionally-sensible fallbacks (e.g. `pickKicker` rolls #10 → on-field #9 → any on-field back → any on-field player), so a sin-binned / sent-off / injured player can never be chosen for the role. The bare `attackTeam.players.find(p => p.id === SLOT.FLY_HALF)` pattern is wrong because it can return the binned player; use the helpers.

**Spawn-time stat pipeline.** `applyStarBoost(team)` in `src/team/applyStarBoost.ts` runs once at app start (single ingest point: `src/main.ts:59-62`, before `teamProfile.init()` or any `RawTeamInput` consumer). The determinism harnesses (`scripts/checkDeterminism.ts`, `scripts/checkSeasonDeterminism.ts`) and `scripts/telemetry.ts` apply the same boost so the verification chain sees the same league the player does. Three deterministic, RNG-free passes per team, in order:

1. **Tier calibration** (`applyTierCalibration`). Each non-star, non-irrelevant baseStat gets an additive shift from `TIER_CALIBRATION` (`src/engine/balance/rating.ts`): starting XV `+10`, bench `+3`, wider `squad` `-5`. Stars get `shift = 0`. Irrelevant stats — `IRRELEVANT_STATS[position]` (forwards' kicking, backs' setPiece) — are clamped to `STAR_BOOST.irrelevantStatMax` (15). Per-stat ceilings (`LEAGUE_STAT_CEILINGS`, currently `pace: 95`) and per-player overrides (`PLAYER_STAT_OVERRIDES` — e.g. Arundell 99, Slade 85) bound the shift via `statCap()`.
2. **Per-star boost** (`boostStar`). For each entry in `team.stars[]`, match by full name and floor `indexHigh` stats to `indexHighMin` (95) or `topIndexHighMin` (97 when `suggestedRating ≥ topThreshold` of 90), floor others to `otherStatMin` (78), then iterate the highest-position-weighted non-capped stat by +1 until OVR ≥ `suggestedRating + targetOffset` (3). Stars supersede tier calibration because every star floor is above any non-star tier value.
3. **Per-player overrides** (`applyPlayerOverrides`). `PLAYER_STAT_OVERRIDES` applied verbatim across `players` + `bench` + `squad`. Final word — can exceed `LEAGUE_STAT_CEILINGS` (the Arundell-99 lever).

Adding a new star, league-wide stat cap, or per-player exception is a one-line edit to the relevant constant in `balance/rating.ts` plus (for stars) an entry in `docs/team-data.md` and a regenerate. The on-disk JSONs are never mutated — the boost transforms an in-memory copy at startup. Per-stat caps from steps 1 and 3 are honoured by the boost iteration in step 2 too, so a star with a pace override of 85 has those weight points redistributed to other stats by the iteration loop rather than being clobbered at the end.

**Known limitation:** stars with low `suggestedRating` (≤ 87) can overshoot their target OVR by 1-5 points because the floor pass alone produces an OVR above target and the iteration loop only lifts up. Affects ~9 stars today (Ludlow, Dombrandt, Reffell, etc.) — documented; not a regression. Adding a "scale down to target" step would fix it but the deviation is small enough to defer.

**Career & roster (Phase 1 of the transfer-system roadmap, live as of v2.22a).** `GameState.career` (`src/types/gameState.ts`) holds the persistent senior-squad roster + per-club squad pointers + archived season standings, so the league can span multiple years. Three load-bearing distinctions:
- **`Player.rosterId` is the persistent identity** (globally unique, allocated once by `src/game/rosterSeeder.ts` at the first-ever new-game start or v4-save migration). **`Player.id` remains the matchday slot 1-23** — every match-engine event variant, `RatingEngine`, `StaminaSystem` etc. continue to read it as a slot. Don't conflate the two. Season-scope `SeasonEvent` variants carry `rosterId`; match-scope `MatchEvent` variants carry `id`.
- **The matchday `Team` is built from the roster, not from JSON, via `src/game/rosterTeamBuilder.ts::buildTeamFromRoster(state, teamJson)`.** Team identity (color / name / stadium / `suggestedTactics`) still comes from the JSON; player data (baseStats reflecting accumulated aging, position, dob, contract pass-through) comes from `state.career.roster`. `MatchCoordinator.initPlayer` accepts the threaded `rosterId` and re-attaches it to the matchday Player so post-match `seasonStatsCollector.snapshotMatch(state)` can map back to the persistent record.
- **`applyStarBoost` runs exactly once per session, on the JSON ingest path in `main.ts`** (still gates `teamProfile.init` + provides the seed data for `rosterSeeder`). It is **not idempotent** — re-applying would compound the tier shift. The roster carries the post-boost `baseStats`; aging mutates the roster's `baseStats` directly via `PLAYER_AGED`. A v5+ save bypasses the seeder entirely and restores the saved (already-boosted, already-aged) roster on load.

**Contracts + cap (Phases 2 + 3, live as of v2.36a).** Every roster Player carries `PlayerContract { clubId, expiresOn, annualWage, isMarquee }` + `reputation: number`, seeded once via `src/game/contractSeeder.ts::seedContractFields` from `rngTransfer` (wage = `WAGE_BY_RATING` × `POSITION_SCARCITY` × `WAGE_NOISE`, length age-banded, expiry staggered ~22/38/42 across +1/+2/+3 years). One hand-authored marquee per club via a `Marquee: yes.` annotation in `docs/team-data.md`, translated by `scripts/generateTeamJsons.mjs` into a `contract: { isMarquee: true }` JSON override that `contractSeeder` honours. `ContractsScreen` (Hub → Contracts) is interactive: tap-to-toggle marquee per row (goes through `MARQUEE_DESIGNATED` so the seam stays clean), 3-state cap pill (ok / tight at ≥ 95% / over). Cap usage = Σ(non-marquee wages); the marquee slot is excluded. Effective cap = `SENIOR_CAP + EFFECTIVE_CAP_CREDITS` — the headline £6.4M plus the £1.4M `CAP_CREDITS` dispensation pools (Home Grown + EPS + injury, modelled flat per-club in v1 — per-player HG/EPS tagging deferred). Tuning constants in `src/engine/balance/transfers.ts`.

**End-of-season renewals (Phase 4, live as of v2.36a).** Between the final-round `EndOfSeasonScreen` and rollover, `gameEngine.openRenewalWindow()` emits `MARKET_OPENED(phase: 'renewals')` with one `TransferOffer` per expiring player league-wide (wage = fresh-market rate × `1 - loyaltyDiscount`, length re-derived from current age via `contractSeeder`). `RenewalsScreen` surfaces the player's club's offers with a per-row Renew/Release toggle (defaults to Renew) + live projected cap. On Continue, `closeRenewalWindow(decisions)` resolves every pending offer in one event-batched call: user decisions for the player's club, AI auto-resolve via `src/game/aiTransferDirector.ts::decideAIOffers` (pure, RNG-free greedy by OVR with marquee + effective-cap-target + `aiReleaseRatingFloor` rules) for the rest. Accepted offers fire `OFFER_RESPONDED(accept)` + `CONTRACT_EXTENDED`; rejected fire `OFFER_RESPONDED(reject)` + `CONTRACT_TERMINATED('expired')` (player joins `state.career.freeAgents`, consumed by the signing window that immediately follows). `MARKET_CLOSED` clears `state.career.market`. Tuning: `RENEWAL.loyaltyDiscount` / `aiTargetCapUtilisation` / `aiReleaseRatingFloor` in `balance/transfers.ts`.

**Competitive signings (Phases 5 + 6 + 10, live as of v2.144a).** The signing window is a multi-round competitive market — every Make Offer + AI bid pass creates a `TransferBid` on `state.career.market.bids`; the resolver picks winners by appeal score, not first-come-first-served. `gameEngine.openSigningWindow()` emits `MARKET_OPENED(phase: 'signings')` with one cached `TransferOffer` per free agent + Reg 7 poach candidate (excluding marquees + the player's own squad); the cached wage is the asking-wage every bidder pays, so appeal — not wage size — decides contests. Per Submit round: `submitBid` adds the user's bids (kind inferred from FA / poach eligibility), `runAIBidPass` adds non-human-club bids (`decideAIBids` — re-evaluated fresh each round, `aiReleaseRatingFloor = 70` floor, `AI_SIGN_CAP_TARGET × club.salaryBudget` headroom per club), `runAIRetentionPass` adds auto-retention bids for AI clubs whose own final-year players are under poach attack (renewal-discounted wage). If the user's own players are under attack, `RetentionDecisionScreen` slots in before resolution so the user can submit their own retention bids. `resolveSigningRound` walks rosterId-ascending; per-player winner = highest `appealScore(state, bid, player)` = `squadAvgOvr × OVR_WEIGHT + positionShortage × NEED_WEIGHT + (5.5 - lastSeasonPosition) × AMBITION_WEIGHT + (isCurrentClub ? LOYALTY_BONUS : 0)`; ties break by lower clubId. Winning bid fires `CONTRACT_SIGNED` (FA) / `PRE_AGREEMENT_SIGNED` (poach) / `CONTRACT_EXTENDED` (retention) alongside `BID_RESOLVED { won }` per bid. `SigningResultsScreen` then shows outcomes; Continue loops back to `TransferMarketScreen` when budget headroom + viable candidates remain (`hasViableSigningOptions`), otherwise auto-finishes. Finish (or auto-finish) calls `closeSigningWindow` for a final AI bid + retention + resolution pass (so AI clubs that didn't bid in any round still get to fill squads), then `MARKET_CLOSED` then `rollSeason()`. Pending bid wages are reserved against `clubBudgetUsage` so the hard budget gate fires on the next Make Offer; lost bids release their reservation via `BID_RESOLVED { lost }`. Marquee wages stay excluded from the budget (same accounting as cap).

**Generated supply (Phase 7, live as of v2.43a).** `src/game/personaGenerator.ts::generatePersona(seed, calendarDate)` produces deterministic `Player` records from `rngTransfer` — name from per-nationality `NAME_POOLS` (English / Welsh / Scottish / Irish / French / South Africa / NZ / Australia / Fiji / Argentina, ~15-20 first + last names each), nationality biased by `NATIONALITY_BY_CLUB` for academy grads (3/6 slots English) or uniform for foreign imports, position uniform across all 12 generic positions, dob anchored to season-open year ± `ageBand`, baseStats `targetOverall + N(0, ±12)` clamped 1-99. Wages: academy fixed £20k (RPA rookie rate); imports `WAGE_BY_RATING × POSITION_SCARCITY` rounded to £5k. Fired during `careerRollover.computeRollover` after retirement / aging passes: `ACADEMY_GRADUATED` 2-4 per club in stable id-ascending order, `FOREIGN_IMPORT_ARRIVED` 5-10 per window. Per-rollover roster growth ~32 players (≈ 480 by end of season 2 from a seeded 448). Stable iteration order in `careerRollover` keeps the `rngTransfer` call sequence deterministic.

**Save schema.** `SAVE_VERSION = 18`. v17 saves auto-migrate by back-filling `Player.condition = 100` on every roster entry (training system addition — `backfillRosterSeasonStats` does the back-fill alongside the season-stats one); the top-level `training` field is optional and v17 saves load with it undefined so `TrainingScreen` resolves via `DEFAULT_TRAINING_PLAN`. v16 saves additionally auto-migrate by defaulting `tactics.offloadStrategy` to `'balanced'` (numerically neutral) — older saves predate the cautious/balanced/offload_freely dimension and would otherwise hand the engine `undefined`. v15 saves auto-migrate by defaulting `career.midseasonRejections` to `{}` and accepting the new `MarketState.phase` value `'signings-midseason'` (no historical mid-season cooldowns — the user can immediately re-approach anyone on first load). v14 saves auto-migrate by defaulting `MarketState.bids` to `[]` (Phase 10 competitive signings — any prior in-window signings stay committed via `CONTRACT_SIGNED`). v13 saves additionally default `ClubState.salaryBudget` to the effective cap (no retroactive budget constraint — kicks in at the next rollover via `computeBudgetEvents`) and `career.takeoverHistory` to `[]` (Newcastle Red Bull then re-fires at the next year-1→year-2 transition if applicable). v12 saves additionally default `state.league.playoffs` to undefined (no active bracket) and back-fill `championTeamId: null` on every archived season; `continueGame` then calls `seedPlayoffBracket()` so a save stuck at "all 18 played, no playoffs" auto-seeds and enters the playoff chain. v11 saves additionally default `career.preSeasonStep` to undefined — Squad Builder resumption is purely additive and older saves skip straight to Hub. v10 saves additionally default `homeTries` / `awayTries` on each persisted `FixtureResult` to 0 (pre-v11 rounds were played without try-bonus tracking, so no retroactive bonuses are fabricated). v9 saves additionally back-fill `tactics.defensiveLine = 'hybrid'` (numerically neutral) — older saves predate the blitz/hybrid/drift dimension and would otherwise hand the engine `undefined`. v8 saves additionally default `state.league.teamSeasonStats` to `{}` and back-fill each player's `seasonStats` with zeroes on the v9 fields (carries, metres, line breaks, etc. — see `backfillRosterSeasonStats` in `src/ui/SaveManager.ts`). v7 saves additionally default `pendingMoves` to `[]`; v6 defaults `freeAgents` / `market` to `[]` / `null`; v5 saves run `contractSeeder` over the persisted roster; v4 triggers a fresh roster seed; v3/v2 schedule-only restore; v1 discarded. The persisted state always flows through `CAREER_ARCHIVE_RESTORED` (which accepts optional `freeAgents` + `market` + `pendingMoves` + `teamSeasonStats` + `preSeasonStep` + `playoffs` + `takeoverHistory` + `midseasonRejections`), so the mutation seam holds. Full version table in `docs/game-engine.md` § "Save format".

**Season-stats architecture.** Two parallel accumulators reset on `SEASON_ROLLED_OVER`:

- **`Player.seasonStats`** (`src/types/player.ts`, in `state.career.roster[rosterId]`) — per-player counters summed from `PlayerMatchStats` every fixture: appearances, tries, carries, metresCarried, lineBreaks, defendersBeaten, passes, kicksFromHand/kickMetres/kicksAtGoal/kicksMade, tackles/missedTackles/dominantTackles, turnoversWon, lineoutThrows/Wins/Catches/Steals, scrumPenaltiesWon/Conceded, rucksHit, yellowCards/redCards, plus `ratingSum` (avg match rating = `ratingSum / appearances`).
- **`state.league.teamSeasonStats`** — keyed by teamId. Per-club season aggregates: matchesPlayed, possession/territory/matchSeconds, attack (tries, lineBreaks, carries, metres, defendersBeaten), defence (tacklesAttempted/Made, turnoversWon), kicking, set-piece win rates (own lineouts / scrums), entries 22, discipline (knock-ons, cards). Possession/territory/set-piece pull from `MatchState.stats`; attack/defence/kicking pull from per-player matchStats sums — same split as `scripts/telemetry.ts::sumMatchStats`.

Both are populated by `src/game/seasonStatsCollector.ts::snapshotMatch(state, homeId, awayId)` returning a `MatchSnapshot`; `collectSeasonEvents(snapshot)` produces N+2 events (one `PLAYER_SEASON_STATS_ACCUMULATED` per player who took the field plus two `TEAM_SEASON_STATS_ACCUMULATED` events). The same snapshot flows from the live-match path (`main.ts` → `recordPlayerMatchResult`) and every silent AI fixture (`simulateFixture.ts`), so the player and the AI league populate identically. Read helpers — `playerLeaderboard`, `leaderboardAvgRating`, `teamLeaderboard`, `teamSeasonStat`, `teamPossessionPct`, `teamTerritoryPct`, `averageRating` — live in `src/game/seasonLeaderboards.ts`. At rollover, `careerRollover.computeAwards` snapshots top-3 per category into `ArchivedSeason.leaders: SeasonAwards` (configured via `SEASON_AWARDS.leaderboardSize` in `balance/career.ts`); the existing `topScorerRosterId` / `mvpRosterId` fields are kept as back-compat shortcuts for `leaders.topTries[0]` and `leaders.topRating[0]`.

**Known gap — goal-kicking split.** `PlayerSeasonStats.conversions`, `.penaltiesScored`, `.dropGoals` permanently sit at 0 today because the match engine doesn't tag a goal kick as conversion vs penalty vs drop at the player level. `PlayerMatchStats.kicksAtGoal` and `.kicksMade` carry the lumped totals; the season aggregator surfaces those, but the split fields are reserved. To fill them, the kick resolver would need to emit a `KICK_AT_GOAL_RESULT` event carrying `kind: 'conversion' | 'penalty' | 'drop'` plus the kicker's id, and `applyMatchEvent` would route into per-category counters on `PlayerMatchStats`. Until then leaderboards key off tries / kicksMade lump only.

**Season schedule.** `src/data/fixtures-2025-26.ts` exports `PREMIERSHIP_2025_26: SeasonSchedule` — the canonical 90-fixture list for the 2025/26 Premiership (5 fixtures × 18 rounds, full double round-robin). Each `Fixture` carries its real ISO `date`. `GameCoordinator.newSeason()` and `fromSave()` default to this schedule; `applySeasonEvent`'s `SEASON_INITIALIZED` ingests it directly. `WEEK_ADVANCED` jumps `calendar.date` to the next round's earliest fixture date (so the Autumn Nations and Six Nations breaks appear without special-casing); falls back to `+SEASON_VALUES.weekLengthDays` when no per-round date is present. `calendar.week` is the upcoming round number, not a wall-clock week. **Source of truth is `docs/prem-fixtures-2025-26.md`** — the doc and the TS module must stay in sync when fixtures change. `src/game/fixtures.ts::generateFixtures` (circle-method round-robin) is retained for future random-gen seasons — it produces fixtures with no `date`, so the +7-day fallback applies; not called at season init today, do not delete as dead code.

## Future development (intentionally out of scope today)

Deliberate gaps. The architecture is ready for each; no refactor needed when adding.

### Transfer system extensions (post-Phase-7)
- **All seven roadmap phases are live** (multi-season rollover, read-only contracts, interactive marquee + cap, end-of-season renewals, free-agent signings, Reg 7 cross-Prem poaching, generated academy + foreign supply). Remaining work is refinement, not roadmap: per-player HG/EPS cap tagging (replacing the flat `CAP_CREDITS` model), reputation drift from silverware / squad form, transfer budgets distinct from cap, squad size limits, owner cash, mid-season transfers / loans / buyouts. All would extend the existing `applySeasonEvent` seam, the `rngTransfer` stream, and the `TransferOffer` / `PreAgreement` shapes — no new architectural primitives needed. See **`docs/transfer-system.md`** § "Open implementation questions" for the running list.

### Card system extensions
- **`PenaltyOffence` taxonomy** (`src/types/engine.ts`) covers seven offences as of v2.61a: `breakdown_infringement` / `scrum_infringement` / `high_tackle` (TMO 90%) / `offside_at_ruck` / `obstruction` / `dangerous_cleanout` (TMO 90%) / `not_rolling_away`. Adding another is a 4-step extension: add the union variant in `engine.ts`, give it a row in `OFFENCE_SPEC` (`balance/discipline.ts` — only field today is `tmoTriggerPct`), emit `PENALTY_AWARDED { offence: '<new>', ... }` from the appropriate phase event, and add a `PhaseOutcomeKey` + commentary templates. The `CardHandler` TMO gate is registry-driven so no branch edit there.

### Maul phase
- **Driving-maul follow-ups.** v1 ships defending-side-only outcomes (won / held / collapse-penalty). Attacking-side maul penalties (truck-and-trailer obstruction, accidental offside) are deliberately deferred — they're rare and would add a 4th branch with low signal value. Per-player maul stats (`maulPenaltiesWon` etc.) are not tracked; if they're useful later they need new `PlayerMatchStats` / `PlayerSeasonStats` fields plus a SAVE_VERSION bump for back-fill (same shape as the existing scrum-penalty counters). The "missing forward weakens the pack" mechanic is naturally enforced by `availableForwards` shrinkage in `MaulResolver.packScore` — same sum-not-average trick as the scrum — so no dedicated `SHORT_HANDED.missingForwardMaulPenalty` constant is needed.

### AITacticalDirector card-awareness
- **`AITacticalDirector` (`src/engine/AITacticalDirector.ts`) does not read `state.cards`.** A team a man down (yellow / red_20 / red_full for a back) should arguably flip the AI more defensive (`AI_INTENT_PROTECTING`-style overlay) regardless of score gap. Extension: add a `playerCountTrigger` block to `src/engine/balance/aiDirector.ts` and a card-count check inside `director.evaluate()`. Pure additive — no breaking change to the baseline-vs-intent bundle architecture.

### Compressed-match model (per-tick abstraction)
- **A simulated match is dramatically shorter than a real 80-minute game in terms of resolved events.** Each tick consumes `0.2 + rng(0, 0.8)` game minutes (~0.6 avg) so a full match runs ~130 ticks total. One tick resolves at most one phase — a Carry, Scrum, Lineout, etc. — so per-player matchStats counts (carries, tackles, passes) come out at roughly 10-20% of real-world Premiership volumes (~30 carries/match combined vs ~250 real). This is a deliberate abstraction, not a bug: the engine models the *outcomes* of a phase cluster, not every micro-event inside it. Telemetry sections that report per-match aggregates should be read as "decision-significant events", not literal real-rugby counts. The per-team match-stat ratios (tackle completion %, possession %, lineout/scrum win %) and the score / try / penalty totals remain calibrated against real benchmarks.
- **Planned: commentary on key moments + fast-forward silent simulation between them.** The current engine runs every tick at the same level of detail and (in live mode) narrates each one. A future change would split ticks into two paths: a "key moment" path (try-line carries, set pieces inside the 22, penalties, cards, late-game decisions) that resolves and narrates in full; and a "fast-forward" silent path that batches sub-significant ticks (midfield carries, neutral kicks-and-chases) and emits a compressed summary every few real seconds. This would let us raise the resolved-tick count toward real-rugby volumes (≈ 250 carries / 280 tackles per match) so individual stat lines and leaderboards look proportional, without slowing the live commentary feed. Implementation hook is in `MatchCoordinator.tick()` — gate per-tick `emitEvent` on a "key moment" classifier, keep the silent batching path for everything else. No state-shape change needed; the existing `MatchEvent` taxonomy already covers everything that would fire in either mode.
